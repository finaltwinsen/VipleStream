## Fixes

- **§J.3.e.2.i.10e ROOT-CAUSE FIX** — `kFrucFramesInFlight` 4→2 revert (commit `6b9bb89`, v1.4.8). The v1.4.2 bump from 2→4 was the wrong direction; with N=4 the AVFrame hold cycle was 80 ms, mapping exactly to what users observed as a stable `decodeMeanMs` ~80–100 ms. Reverting to N=2 produced a 5–90 ms bimodal pattern.
- **§J.3.e.2.i.10f Path D** — early AVFrame release via two-submit pattern (commit `9fd8c22`, v1.4.9, final fix). `renderFrame` is split into `m_SlotCopyCmdBuf` (~100 µs) and `m_SlotCmdBuf` (~20 ms) submits on the graphics queue. AVFrame hold cycle drops from 40 ms to 0.2 ms; live `decodeMeanMs` 0.47–30 ms typical, on par with D3D11 / FRUC OFF.
- **§J.3.e.X β.5.3 D-lite** — `inferDim` rounding switched from DOWN to UP to a /128 multiple (commit `9d4c237`, v1.4.1), resolving the RIFE-v4-lite Add_503 layer shape constraint mismatch when `inferDim` was not a /128 multiple.
- **§J.3.e.X β.11** — TRIPLE+ME `EDGE_AWARE_MV_THRESHOLD` 2.0 → 8.0 (commit `fa46a88`), addressing the dynamic-edge 8×8 mosaic artifact. Filters noise to preserve bilinear smoothing; large motion boundaries (>2.83 px diff) still switch to nearest-pick to keep edges sharp.
- **§N.4** — Server `/transfer/poll` non-blocking endpoint bug. `manager::poll(timeout=0)` was misread as "≤0 means use 30 s default" and blocked 30 s; the client's 15 s watchdog always won, killing polling entirely. Restoring true non-blocking semantics unblocked the whole feature.
- **§N.4b** — Server downloads-dir was being written under the SYSTEM profile. The LocalSystem service account calling `SHGetKnownFolderPath(nullptr)` returned `C:\Windows\system32\config\systemprofile\Downloads`, invisible to the user. Switched to using the active console user's token to resolve `C:\Users\<user>\Downloads`.
- **§N.5.bug (b)** — Server `sweep_stale_locked`: file transfer commands auto-release their lock 60 s after a client crash / disconnection (preventing the next client's transfer from getting stuck). **(a)** Conscrypt post-handshake `TLSV1_ALERT_INTERNAL_ERROR` is worked around client-side via the Supplier pattern (a fresh wrapped OkHttpClient per call, avoiding cached SSL state corruption).
- **§M.1.f client-side** — moonlight-qt `session.cpp` splits `shouldQuit` into `shouldNotifyServerCancel` (graceful exit always sends `/cancel`) and `shouldQuitClientApp` (client-app exit still gated by the `quitAppAfter` preference). The Android leg in `Game.java stopConnection()` does the same: before `conn.stop()`, build a transient `NvHTTP` from `xferHost / xferHttpsPort / xferServerCert` and call `quitApp()` inside a best-effort try-catch.
- **§M.1.f server-side defensive race revert** — v1.4.39 reverts the `rtsp.cpp` `remove()` / `clear()` `proc.clear_owner_uuid()` hook that fired when `session_count` hit zero. Host-log evidence showed the hook firing during normal stream setup (14:02:50, between launch at 14:02:47 and RTSP session insert at 14:02:50.197), wiping the just-set owner. The `clear_owner_uuid()` public method is retained for a future `last_activity` timeout mechanism.

## Upgrade

- Path β (Native RIFE Vulkan) users: this release **automatically** lowers the `inferDim` default from 256 to 128. RTX 4070+ users wanting 256 quality can switch back via Settings → Video → Frame Interpolation. Mid-range GPUs (RTX 3060 and below) should keep 128 to stay within the per-frame budget.
- Users sharing a host across multiple paired devices: §M.1 + §M.1.f fixes apply automatically with no configuration. The Web UI now includes a session dashboard listing the current owner plus a force-disconnect button.
- In-stream file transfer: use the tray menu "Send file to client" on the Windows host to pick a file → it lands in the client's Downloads. While streaming, the client-side web UI at `https://<host>:47990/transfer` provides Receive-from-client functionality.
- Linux server installers: this release fixes the `fs_picker` linker gap, so the `.deb` installs cleanly and the service starts (but the tray "Send to client" action is currently a no-op — real zenity integration is tracked under §N.7).

## Major technical milestones

### §J.3.e.2.i.10 Path D — Vulkan + FRUC 80–100 ms regression root cause + final fix

**Symptom**: After v1.4.0, live testing with `RS_VULKAN` + Native RIFE FRUC ON consistently showed `decodeMeanMs` ~80–100 ms (sub-millisecond on D3D11 / FRUC OFF) — an obvious regression.

**Root cause (v1.4.8 commit `6b9bb89`)**: v1.4.2 (`234d2aa`) bumped `kFrucFramesInFlight` from 2 to 4 to widen CPU pipelining. But Vulkan's single graphics-queue command buffer execution is FIFO — adding slots only expands CPU pipelining, not GPU throughput. Each slot still runs the full chain (~20 ms) before signalling `vkf->sem[0]@V+1` to release the AVFrame back to the FFmpeg pool, so **AVFrame hold cycle = N × chain_time**. N=4 → 80 ms hold cycle, exactly matching the observed `decodeMeanMs`. Reverting to N=2 brought the hold cycle to 40 ms and `decodeMeanMs` to a 5–90 ms bimodal pattern (5–15 ms in good stretches).

**Final fix Path D (v1.4.9 commit `9fd8c22`)**: split `renderFrame` into two graphics-queue submits.
- **Submit 1** (`m_SlotCopyCmdBuf`, ~100 µs): does only the image→buffer copy and signals `vkf->sem[0]@V+1`. The AVFrame is released back to the FFmpeg pool within ~100 µs.
- **Submit 2** (`m_SlotCmdBuf`, ~20 ms): runs the full RIFE chain + warp + present, without touching `vkf`.

AVFrame hold cycle drops from 40 ms to 0.2 ms. **Live test: `decodeMeanMs` 0.47–30 ms typical, sub-millisecond often, on par with v339 / D3D11 / FRUC OFF.** The enabling change was §B-quality (d) on 2026-05-06, which switched DUAL+FRUC real-frame display to `m_FrucCurrRgbBuf` via `m_RealCurrRgbDescSet` (no longer sampling `vkf->img[0]` through a ycbcr sampler), so `vkf->img[0]` is now only used for ~100 µs at the start of the chain.

### §J.3.e.X Path β quality + latency cleanup

- **β.5.2 Catmull-Rom bicubic upsampling for flow + mask** (commit `a8b3204`, v1.4.1) — score 0.95 → 0.97–0.98; warp shader chain +0.4 ms, still within budget.
- **β.5.3 D-lite UP-rounding `inferDim`** — RIFE-v4-lite Add_503 layer shape alignment fix.

### §J.3.e.Y 4Y.7 C.1 Conv→BinaryOp(Mul) fusion

40 fuseable pairs absorbed into the Conv shader epilogue; chain −2.6 ms (13.8 → 11.2 ms @ 256×128 baseline). One of the highest-ROI optimisations on the native RIFE path outside the Tensor Core route.

### §H.3 `inferDim` default 256 → 128 + UI honesty

On a mid-range GPU (RTX 3060 Laptop), `inferDim=256` gives a ~20 ms chain — over the 16.7 ms 60 fps slot — pushing dual-present from a 120 target down to 45.34 with p99 28 ms judder. `inferDim=128` chain ~10 ms fits the budget at 76.69 fps. The default is now 128, with 256 reserved as opt-in for RTX 4070+. `SettingsView.qml` ComboBox text and the RIFE checkbox tooltip have been updated to match the measured numbers.

### §M.1 Multi-user ownership guards + Web UI session dashboard

Fixes the "multi-user shared server: client A accidentally kills client B's Steam game" bug. `proc::proc._owner_uuid` is a per-launch lock — client A's `/launch` / `/resume` / `/cancel` cannot affect a session client B is currently using. The Web UI adds a session dashboard listing the current owner's UUID, name, start time, and a force-disconnect button.

### §M.1.f quit-key combo releases server-side ownership automatically

**Root cause**: in moonlight-qt `session.cpp:1395-1397`, the `shouldQuit` boolean bundled two concerns — "notify server via /cancel" and "client app exit" — into a single condition (`!m_UnexpectedTermination && m_Preferences->quitAppAfter`). The `quitAppAfter` preference defaults to false, so the quit-key path never sent `/cancel` to the server, `proc::proc._owner_uuid` was never released, and the next paired device's `/launch` got a 503 deny.

**Fix**:
- **moonlight-qt (v1.4.38)**: split `shouldQuit` into `shouldNotifyServerCancel` (`!m_UnexpectedTermination`, always send `/cancel` on graceful exit) and `shouldQuitClientApp` (`!m_UnexpectedTermination && quitAppAfter`, controls the client app exit behaviour as before).
- **moonlight-android (v1.4.40)**: in `Game.java stopConnection()`, before `conn.stop()`, build a transient `NvHTTP` using the cached `xferHost / xferHttpsPort / xferServerCert` (the same auth handle file transfer uses) and call `quitApp()` inside a best-effort try-catch.
- **server-side `clear_owner_uuid()` method** (v1.4.38) retained for a future `last_activity` timeout mechanism.
- **server-side rtsp.cpp defensive hook reverted** (v1.4.39) — the original `remove()` / `clear()` session_count==0 → `proc.clear_owner_uuid()` hook lost a race with normal stream setup (the hook fired between launch and RTSP session insert, wiping the just-set owner). See §M.1.f.2 follow-up in known issues.

### §N In-stream bidirectional file transfer

Full server + moonlight-qt + moonlight-android trio:

- **Server** (v1.4.14 → v1.4.32, `Sunshine/src/file_transfer.{h,cpp}` + `nvhttp.cpp` + `confighttp.cpp` + `system_tray.cpp`): per-paired-client command queue + token registry + chunked I/O + path sanitisation + active-user Downloads dir resolver (`WTSGetActiveConsoleSessionId` + `WTSQueryUserToken`). 5 HTTPS endpoints (`/transfer/poll` / `/transfer/result` / `/transfer/blob` / `/transfer/progress`) + 4 confighttp endpoints (`/transfer` HTML / `/transfer/listing/latest` / `/transfer/pull` / `/transfer/progress-proxy`). The tray menu gains "Send file to client" / "Receive file from client" entries. Windows native picker at `src/platform/windows/fs_picker.cpp` (an `IFileOpenDialog` COM wrapper). The Receive UI is a file-browser-style interface: path input, breadcrumbs, Up/Home, click-into folders, "Pull as zip" for folders (PowerShell `Compress-Archive` on Windows, `zip` on Linux).
- **moonlight-qt client** (v1.4.15 → v1.4.33, `streaming/transfer/filetransferclient.{h,cpp}`): 2 s polling + LIST_DIR / DOWNLOAD_TO_CLIENT / UPLOAD_FROM_CLIENT / CANCEL dispatcher. **Dedicated QThread + Qt event loop** — `Session::exec()` runs the SDL main loop and starves the main-thread Qt event loop, so QTimer / QNetworkAccessManager must live on a separate thread. Stream-end teardown clears the QPointer before `abort()` to avoid a same-thread synchronous `finished`-signal callback null-deref. SSL `onSslErrors` is connected to the pinned `m_Computer->serverCert`; if it matches we `ignoreSslErrors`. After each transfer completes, `m_Nam->clearAccessCache()` refreshes the pool to avoid half-closed socket reuse. Folder zip uploads invoke `Compress-Archive` (Windows) or `zip` (Linux) as a subprocess and clean up the temp file afterwards.
- **moonlight-android client** (v1.4.19 onward, `transfer/FileTransferClient.java`): OkHttp polling + `MediaStore.Downloads` scoped-storage read/write + streaming `RequestBody`. v1.4.38 §N UX polish replaced the toast with an in-stream overlay (bottom-left yellow-green text, 0–100% visible throughout, auto-hides 3 s after completion or failure). Conscrypt SSL state corruption is worked around with the Supplier pattern — a fresh wrapped `OkHttpClient` per call (v1.4.34 §N.5.bug client-side workaround).

## Known issues

- **§N.5.linux mouse + overlay** — on the Linux client (Ubuntu VM AppImage) ↔ Windows host path, mouse-input forwarding does not work (SDL2 mouse capture + xcb session interaction) and overlay characters render as garbled glyphs (AppImage does not bundle CJK fonts). Keyboard input and the file transfer endpoint remain usable. Deferred to v1.4.41+.
- **§N.5.bug Scenario A** — the underlying source of the Conscrypt `TLSV1_ALERT_INTERNAL_ERROR` alert has not been pinpointed (the client-side Supplier pattern works around it); the SimpleWeb / boost-asio handler entry/exit/try-catch diagnostic is still pending.
- **§N.7 real zenity fs_picker** — v1.4.40 ships only a stub (returns `nullopt`); the Linux server tray's "Send to client" action is currently a no-op. A native Linux GUI session integration with zenity subprocess is still TODO.
- **§M.1.f.2 abnormal disconnect ownership timeout** — client crash / WiFi drop / power off paths never send `/cancel`, so the server's `_owner_uuid` can still get stuck. The defensive RTSP hook was reverted in v1.4.39 due to the race; a `last_activity` timestamp + idle reconcile mechanism (not hook-on-teardown) is the planned follow-up.
- **§J.3.e.X β.11** — threshold 8.0 is hypothesis-1 baseline, awaiting user testing to decide whether to keep, try 16 (might still mosaic), revert to 4 (edges may blur), or convert to a push-constant runtime tunable. Other hypotheses (luma-gap backstop / big-motion bias / sub-pixel precision) remain on hold.
- **Path β TRIPLE 60→180 + Native RIFE coexistence performance** — β.9 Phase 1 (mutual-exclusion removal + 2× RIFE inference chain) has shipped; β.9 Phase 2 (graph-executor timestep-shared front-end split, cutting chain × 2 from 200% to ~130%) is still active to make the chain @ 256×128 fit the 16.7 ms 60 fps slot.
- **§J.3.e.Y 4Y.6 Tensor Core path** — off by default (−70× precision regression). Enable with `VIPLE_RIFE_VK_COOPMAT=1`.
